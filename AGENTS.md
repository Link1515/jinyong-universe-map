# 專案說明

這是一個**以前端為主的金庸小說人物知識圖譜專案**。  
目標是將金庸小說中的主要角色整理成可視化的**人物關聯圖譜**，讓使用者可以直觀探索角色之間的各種關係。

專案核心價值：

- 以圖譜方式呈現人物網絡，而不是傳統列表或條目式資料
- 強調「角色與角色之間的關聯」
- 提供易於探索、篩選、搜尋與切換作品的互動體驗
- 介面需清楚、美觀、可擴充，便於後續新增人物、作品與關係類型

---

# 專案定位

- 這是**前端專案**
- 以 JSON 資料驅動方式開發
- 以桌面瀏覽體驗優先，同時兼顧行動裝置的基本可用性

# 關係範圍

人物之間的關聯至少需支援以下類型：

- 血緣關係
  - 父母 / 子女
  - 兄弟姊妹
  - 祖孫
  - 親屬
- 感情 / 婚姻關係
  - 戀人
  - 夫妻
  - 曖昧 / 單戀（若資料有明確依據）
- 師承關係
  - 師父 / 徒弟
  - 同門
- 組織關係
  - 幫會
  - 門派
  - 教派
  - 朝廷 / 官府
  - 其他勢力
- 結義關係
  - 結拜兄弟 / 姐妹
- 對立關係
  - 仇敵
  - 宿敵
  - 追殺 / 被追殺
- 朋友 / 同盟關係
- 主僕 / 隨從關係
- 其他重要特殊關係
  - 收養
  - 救命恩人
  - 競爭者
  - 傳功 / 武學傳承
  - 重要事件中的共同參與者

要求：

- 關係需具備**類型**
- 關係可視需要支援**方向性**（例如師徒、父子）
- 關係可支援**附註**，說明背景或特殊情節
- 關係可支援**來源作品**與必要的情節註記
- 同一對角色之間可能同時存在多種關係，資料模型必須允許多重關係並存

# 核心功能需求

## 1. 人物關聯圖譜視覺化

- 使用關聯圖 / 網路圖呈現人物節點與邊
- 節點代表人物
- 邊代表人物之間的關係
- 不同關係類型應有明顯視覺區別，例如：
  - 顏色
  - 線條樣式
  - icon / 標籤
- 圖譜需支援基本互動：
  - 拖曳節點
  - 縮放
  - 平移
  - 點擊人物查看詳細資訊
  - 點擊關係查看關係說明

## 2. 作品切換

- 可依作品切換查看各小說人物圖譜
- 可提供：
  - 單作品視圖
  - 全作品總覽視圖（進階，可後續實作）
- 切換作品時，圖譜與側邊資訊需同步更新

## 3. 人物搜尋

- 可依角色名稱搜尋
- 搜尋結果應能：
  - 高亮對應人物
  - 自動聚焦到該節點
  - 開啟人物資訊卡

## 4. 關係篩選

使用者可依關係類型篩選圖譜顯示內容，例如：

- 只看血緣
- 只看門派 / 幫會
- 只看師徒
- 只看感情 / 婚姻
- 顯示全部

## 5. 人物詳細資訊面板

點擊人物後，顯示側邊欄或彈窗，至少包含：

- 姓名
- 所屬作品
- 別名 / 稱號（若有）
- 性別（可選）
- 身分 / 門派 / 幫會
- 角色簡介
- 重要關係列表
- 可從資訊面板再次跳轉到其他相關人物

## 6. 關係詳細資訊

點擊兩人物之間的連線後，可顯示：

- 關係類型
- 關係雙方
- 關係說明
- 所屬作品
- 補充備註（例如事件背景）

# 資料模型建議

請以可擴充、易維護為原則設計資料結構。建議至少包含以下概念：

## Character

- id
- name
- aliases
- novel
- gender
- faction / sect / organization
- title
- description
- tags

## Relationship

- id
- source
- target
- type
- label
- description
- directed
- novel
- weight / importance（可選，用於視覺強度）
- metadata（可選）

## Novel

- id
- name
- era / background（可選）
- description（可選）

要求：

- 資料應盡量正規化
- 避免把顯示邏輯硬編碼在資料中
- 前端展示層與資料層應適度分離
- 建議使用靜態 JSON / TypeScript 型別定義來管理資料

---

# UI / UX 要求

- 介面風格應簡潔、清楚，避免過度花俏影響閱讀
- 資訊密度要高但不混亂
- 大量人物關係時，仍需保持可讀性
- 節點與關係在 hover / active / selected 狀態下需有明確反饋
- 圖例（legend）必須清楚說明不同顏色或線條代表的關係類型
- 在節點數較多時，需考慮效能與畫面擁擠問題
- 行動裝置至少要能查看人物資訊與進行基本操作

---

# 技術實作原則

- 優先採用現代前端框架與元件化架構
- 程式碼需具備良好可讀性與模組化
- 型別明確，避免鬆散資料結構
- 保持可測試性與可維護性
- 需考慮未來擴充：
  - 新增作品
  - 新增關係類型
  - 新增角色屬性
  - 串接 API / 後端資料來源

如需選擇圖譜渲染方案，請以以下原則評估：

- 是否適合人物網路圖互動
- 是否支援節點拖曳、縮放、事件處理
- 是否容易與前端框架整合
- 是否有足夠社群資源與維護性
- 是否能處理中等規模節點數量而不至於卡頓

# 開發要求

Codex 在開發時請遵守以下原則：

1. 先建立清楚的專案結構與資料模型，再開始堆疊 UI
2. 優先完成可用的 MVP：
   - 作品切換
   - 人物圖譜
   - 人物搜尋
   - 關係篩選
   - 人物詳細資訊面板
3. 先以少量代表性角色驗證互動流程，再擴充完整資料
4. 不要把人物資料直接寫死在元件內，應集中於獨立資料檔
5. 將「資料處理邏輯」、「圖譜渲染邏輯」、「UI 控制邏輯」拆分
6. 為關鍵資料型別、元件 props、狀態管理建立清楚定義
7. 實作時應保留後續加入：
   - 多作品交叉關聯
   - 時間線
   - 陣營視圖
   - 人物群組聚合
   - 關係強度分析
     的擴充空間

---

# MVP 驗收標準

至少需達成以下條件：

- 可切換不同金庸作品
- 每部作品可顯示主要角色的關聯圖
- 使用者可搜尋人物
- 使用者可依關係類型篩選
- 點擊人物可查看詳細資料
- 點擊關係可查看關係說明
- 畫面在常見桌面瀏覽器上可正常操作
- 資料結構清楚，便於擴充更多角色與關係

# 額外注意事項

- 人物與關係資訊需盡量忠於金庸原著設定
- 若某些關係存在版本差異、模糊性或爭議，請在資料中保留備註能力
- 命名需一致，避免同一角色出現多個不一致名稱
- 關係類型應集中管理，避免自由字串造成混亂
- 實作上應優先確保資料正確性與互動可用性，再追求視覺特效
- 一本小說中的人物可能與其他小說中的人物有關連
